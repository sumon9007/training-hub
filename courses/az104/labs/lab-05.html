Lab 05 🟡 Intermediate ⏱ 30 min

Intersite Connectivity

Configure VNet peering and test connectivity between virtual networks

Module 5 — Intersite Connectivity

Lab Objectives

  • ✓Create two virtual networks with non-overlapping address spaces
  • ✓Configure bidirectional VNet peering
  • ✓Deploy test VMs in each VNet
  • ✓Validate cross-VNet connectivity with ping and traceroute

Prerequisites

  • •Azure subscription with Contributor role
  • •Resource group az104-rg5 in East US

Lab Steps

Architecture Diagram
1

Create two virtual networks

Portal

Create two VNets (repeat for each). First: Name az104-vnet1, address space 10.51.0.0/22, add subnet subnet0 10.51.0.0/24. Second: Name az104-vnet2, address space 10.52.0.0/22, add subnet subnet0 10.52.0.0/24. Both in Resource group az104-rg5, Region East US.

2

Create VNet peering (both directions)

Portal

Open az104-vnet1 → Peerings → + Add. Set peering link name (this VNet) vnet1-to-vnet2, Remote VNet az104-vnet2, peering link name (remote) vnet2-to-vnet1. Enable Allow virtual network access on both sides. Click Add — Azure creates both directions simultaneously.

3

Deploy test VMs

Portal

Deploy a small VM (Standard_B1s) in each VNet subnet. Use Linux (Ubuntu 22.04) with SSH public key auth. Note both private IP addresses.

4

Test connectivity via serial console

Portal

Open the VM in VNet1 via Serial console (or SSH). Ping the private IP of the VM in VNet2.

# On VM1 (10.51.0.x):
ping 10.52.0.4 -c 4
traceroute 10.52.0.4
5

Verify peering state

Portal

Open az104-vnet1 → Peerings. The peering vnet1-to-vnet2 should show Connected. Open az104-vnet2 → Peerings to confirm the reverse link vnet2-to-vnet1 is also Connected.

Validation Checklist

  • ✓Both peerings show Connected state.
  • ✓Ping from VM1 to VM2's private IP succeeds with <1ms latency.
  • ✓Traceroute shows direct path (1 hop) between peered VNets.

Cleanup Resources

Run after the lab to avoid ongoing charges.

az group delete --name az104-rg5 --yes --no-wait
