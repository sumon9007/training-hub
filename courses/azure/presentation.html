Module 01 · Cloud Concepts

What is Cloud Computing?

The on-demand delivery of compute, storage, and networking over the internet — with pay-as-you-go pricing and no upfront capital investment in hardware.

🖥️

Compute

Virtual machines, containers, and serverless functions — provisioned in seconds

💾

Storage

Blob, file, queue, and disk storage — petabyte-scale, globally durable

🌐

Networking

Virtual networks, load balancers, and CDN — configurable via API or portal

Emphasise that cloud computing is not simply "someone else's server." The key differentiator is on-demand self-service and the elasticity that follows. Azure provides all three resource pillars — compute, storage, and networking — as metered services accessible over the internet.

Module 01 · Cloud Concepts

The Five Core Characteristics

Defined by NIST — these are the properties that distinguish cloud from traditional hosting.

🙋

On-demand self-service

Provision compute unilaterally, without human interaction from the provider

📡

Broad network access

Resources available over the internet via standard mechanisms from any device

🏊

Resource pooling

Multi-tenant model — physical resources shared, logically isolated per customer

📐

Rapid elasticity

Scale out or in automatically — to the consumer, capacity appears unlimited

📊

Measured service

Usage is metered, monitored, and reported — you pay only for what you consume

These five NIST characteristics are occasionally tested directly on AZ-900. The most commonly confused pair is rapid elasticity (automatic scaling) versus on-demand self-service (self-provisioning without calling the provider). Walk through a concrete example: spinning up an Azure VM from the portal with no call to Microsoft illustrates both characteristics at once.

Module 01 · Cloud Concepts

Business Benefits of Cloud Adoption

Availability & Reliability

  • High availability — SLA-backed uptime (Azure VMs: up to 99.99% with Availability Zones)
  • Fault tolerance — redundant components absorb failures with no user impact
  • Disaster recovery — geo-replicated backups and failover across Azure regions
  • Geo-distribution — deploy close to users in 60+ regions worldwide

Speed & Scale

  • Scalability — add capacity vertically (bigger VM) or horizontally (more instances)
  • Elasticity — scale out during peaks, scale in during off-hours automatically
  • Agility — deploy new apps in minutes; experiment and iterate rapidly
  • CapEx → OpEx — replace upfront hardware spend with predictable monthly billing
🎯 Exam tip: scalability = planned growth; elasticity = automatic scaling in response to real-time demand. The AZ-900 exam distinguishes these terms.
The exam frequently presents scenarios that ask you to identify which benefit applies. Fault tolerance and high availability are often confused — HA means the system stays up; fault tolerance means it stays up even when components fail. Disaster recovery is a separate concept covering recovery from a regional outage, not just a failed component.

Module 01 · Cloud Concepts

Fault Tolerance & Distributed Infrastructure

Azure distributes workloads across redundant physical infrastructure to eliminate single points of failure at every layer.

🏢

Availability Zones

Physically separate datacenters within one region — independent power, cooling, and networking

🌍

Region Pairs

Every Azure region is paired with another ≥300 miles away — updates are staggered, data stays in-country

🔁

Replication

Azure Storage replicates data 3× within a zone and optionally across zones or regions (LRS → ZRS → GRS)

🎯 Exam tip: deploying a VM across two Availability Zones raises its SLA from 99.9% (single VM with Premium SSD) to 99.99% — a common exam scenario.
Walk through the failure domain concept: a rack power failure inside one AZ does not affect the other AZs. Region pairs are key for compliance and disaster recovery — Azure promises to recover at least one region in each pair first during a regional outage. The replication acronyms (LRS, ZRS, GRS, RA-GRS) appear later in Module 2 but introduce them here so attendees have the mental model.

Lesson 01 · Scenario

A retail company runs a shopping website that experiences a tenfold spike in traffic every November during a sale period. For the rest of the year, the site runs at 10% of that peak load. The company wants to minimise infrastructure costs while ensuring the site remains responsive during the spike. Which cloud characteristic most directly addresses this requirement?

A High availability — deploying the site across multiple Availability Zones
B Geo-distribution — hosting the site in multiple Azure regions worldwide
C Elasticity — automatically scaling resources out during the peak and back in afterwards
D Scalability — pre-provisioning a larger VM size to handle the November traffic
✓ C is correct — Elasticity is the ability to automatically increase resources in response to real-time demand and release them when demand drops, which directly matches the spike-and-idle pattern and minimises cost.
  • A — High availability improves uptime and fault tolerance but does not automatically adjust resource quantity to match demand or reduce cost during off-peak periods
  • B — Geo-distribution reduces latency for global users; it does not address cost optimisation for a seasonal traffic pattern
  • D — Scalability describes planned capacity growth (e.g. moving to a larger VM); it is a manual, pre-provisioned change and would over-spend during the 11 off-peak months
Give attendees 30–45 seconds to choose before revealing the answer. The key teaching point is the elasticity-versus-scalability distinction — both involve handling more load, but elasticity is automatic and bidirectional while scalability is deliberate and typically one-directional. This distinction appears in multiple AZ-900 exam questions.

Module 01 · Cloud Concepts

CapEx vs OpEx

CapEx — Capital Expenditure

  • Large upfront investment in physical hardware
  • Depreciated on the balance sheet over 3–5 years
  • Fixed capacity — you own what you bought
  • You are responsible for maintenance and refresh cycles
  • Requires capital budget approval in advance

OpEx — Operational Expenditure

  • Pay for services as you consume them — no upfront cost
  • Deducted as a business expense in the same fiscal year
  • Elastic capacity — scale up or down in minutes
  • Provider owns hardware; you pay for outcomes
  • Comes from operating budget — faster approval cycle
🎯 Exam tip: Cloud computing shifts IT spend from CapEx (buy a server) to OpEx (pay per hour). Both models can co-exist in hybrid scenarios.
Use a concrete analogy: buying a car (CapEx) vs. renting one hourly (OpEx). The depreciation point is often misunderstood — CapEx sits on the balance sheet and reduces profit over several years, whereas OpEx hits the P&L immediately. Finance teams often prefer OpEx because it's predictable on a rolling basis and avoids large approval gates.

Module 01 · Cloud Concepts

The Consumption-Based Model

Azure charges only for what you use — compute time, gigabytes stored, data transferred. If you stop a VM, billing stops. There is no charge for reserved but idle capacity.

🔢

Metered usage

Every resource emits usage metrics every second — cost accumulates like a utility meter, not a flat monthly fee

📉

No idle waste

Deallocate a VM outside business hours and pay only for storage — Azure Example: D2s v5 VM ~$0.096/hr in East US

🔖

Pricing tiers

Pay-as-you-go, Reserved Instances (1- or 3-year), and Spot pricing offer 0%, up to 72%, and up to 90% savings respectively

Highlight that "consumption-based" is what makes cloud OpEx in practice — the bill directly reflects usage. Mention that Azure Cost Management + Billing lets you set budgets and alerts so teams don't overspend. The pricing tier examples are real East US figures as of 2026 — they give attendees a concrete anchor.

Total Cost of Ownership (TCO) & ROI

On-Premises TCO Includes

  • Hardware purchase and refresh (every 3–5 years)
  • Data centre space, power, and cooling (~40% of hardware cost)
  • Network switches, cabling, and load balancers
  • Staff time: OS patching, firmware, capacity planning
  • Software licences and support contracts
  • Disaster recovery site (often doubles costs)

Cloud ROI Levers

  • Eliminate data centre capital refresh cycles
  • Reduce ops headcount — automation replaces manual tasks
  • Faster time to market → earlier revenue recognition
  • Right-size continuously — no stranded capacity
  • Azure Hybrid Benefit reuses existing Windows Server licences
  • Use the Azure TCO Calculator (free, at azure.microsoft.com/tco) for board-ready estimates
🎯 Exam tip: The Azure TCO Calculator compares on-premises costs with Azure costs. The Azure Pricing Calculator estimates Azure-only costs for a proposed architecture.
Students with on-prem backgrounds are often surprised by the full TCO picture — power and cooling alone can equal 40% of the hardware cost per year. Walk through the TCO Calculator URL live if you have internet access. Stress the distinction between the TCO Calculator (on-prem vs cloud comparison) and the Pricing Calculator (Azure cost estimates) — the exam tests this specifically.

Module 01 · Cloud Concepts

Capacity Forecasting & Risk Reduction

On-premises infrastructure forces a binary choice: under-provision (performance risk) or over-provision (wasted spend). Cloud eliminates this dilemma.

📊

Elastic scaling

Azure Virtual Machine Scale Sets and App Service auto-scale add instances in response to CPU or queue depth — no manual pre-ordering

🌐

Global reach, on demand

Spin up resources in a new Azure region in minutes for a product launch — no 12-week hardware lead time

🛡️

Reduced business risk

Pay-as-you-go means a failed experiment costs hours of compute, not a sunk hardware investment of tens of thousands of dollars

The forecasting story resonates strongly with project managers in the room — they've lived through months-long procurement cycles. Use the example of a retail company predicting Black Friday traffic: on-prem means buying for peak and wasting 11 months of capacity; Azure means scaling to peak and scaling back down within hours. This is one of the most compelling cloud value propositions beyond pure cost.

Lesson 02 · Scenario

A retail company currently owns three on-premises servers that handle its e-commerce website. The servers are utilised at roughly 20% capacity most of the year but reach 95% capacity every November and December. The CFO wants to eliminate unpredictable capital expenditure and reduce idle infrastructure costs. Which cloud financial model and pricing approach best meets these requirements?

A Migrate to Azure using 3-year Reserved Instances sized for peak load
B Migrate to Azure using pay-as-you-go consumption pricing and auto-scale for seasonal peaks
C Lease new on-premises servers under a hardware-as-a-service contract
D Move to Azure and purchase Reserved Instances sized for average (20%) utilisation
✓ B is correct — Pay-as-you-go is a consumption-based OpEx model that eliminates upfront capital spend and charges only for resources used; combined with auto-scale, the company pays for peak capacity only during November–December and reduces costs during the remaining 10 months.
  • A — 3-year Reserved Instances sized for peak commit the company to a fixed capacity year-round, preserving the same idle-cost problem as on-premises and introducing upfront financial commitment
  • C — Hardware-as-a-service is still a fixed-capacity, CapEx-adjacent model; it does not provide elastic scaling or eliminate idle spend during low-demand periods
  • D — Reservations sized at average utilisation would be insufficient during peak season, causing performance degradation; Reserved Instances also require a 1- or 3-year commitment, reintroducing financial lock-in
Give attendees 45 seconds to read the scenario and select an answer before revealing. The key teaching moment is that Reserved Instances are a cost-optimisation tool for predictable, steady-state workloads — not for highly seasonal ones. Emphasise that pay-as-you-go plus auto-scale is the textbook answer when demand is spiky and unpredictable. The CFO's goal of eliminating CapEx is the signal that points directly to the consumption-based OpEx model.

Module 01 · Cloud Concepts

IaaS, PaaS, SaaS — The Three Core Models

Cloud services are delivered at different levels of abstraction — the higher the model, the less infrastructure you manage.

🏗️

IaaS

Rent virtualised compute, storage, and networking. You manage the OS and above. Example: Azure VMs

🧰

PaaS

Deploy apps on a managed platform — OS, runtime, and patching handled by Azure. Example: Azure App Service

☁️

SaaS

Use fully managed applications over the internet — no infrastructure or platform ownership. Example: Microsoft 365

The key mental model is a continuum of control vs. convenience. IaaS gives maximum control but maximum responsibility. SaaS gives maximum convenience but least control. PaaS sits in the middle and is the sweet spot for most modern application development. Stress that Azure VM is IaaS, App Service is PaaS, and Microsoft 365 is SaaS — these exact examples appear on the exam.

Module 01 · Cloud Concepts

FaaS & Serverless — Beyond PaaS

Function as a Service (FaaS)

  • Run individual functions triggered by events
  • No server provisioning — ever
  • Billed per execution (millisecond granularity)
  • Scale to zero when idle — no idle cost
  • Azure example: Azure Functions

Serverless Characteristics

  • Abstraction over ALL infrastructure concerns
  • Event-driven execution model
  • Automatic, instantaneous scaling
  • Consumption-based pricing only
  • Also includes: Azure Logic Apps, Event Grid
🎯 Exam tip: "Serverless" does not mean no servers exist — it means you don't provision, manage, or pay for idle servers. Azure Functions is the canonical AZ-900 serverless example.
FaaS is a subset of PaaS but with a fundamentally different billing and scaling model. The exam often presents serverless as a distractor — make sure attendees understand that serverless means event-driven and consumption-billed, not literally server-free. Azure Functions can run on a Consumption plan (true serverless) or a dedicated plan (then it behaves more like PaaS).

Module 01 · Cloud Concepts

Choosing the Right Service Model

Pick IaaS when…

  • You need full OS-level control or custom kernel config
  • Migrating existing on-prem VMs (lift-and-shift)
  • Running software that requires a specific OS version
  • Third-party licensing tied to physical/virtual CPU count

Pick PaaS when…

  • Building or deploying a web app, API, or mobile backend
  • You want automatic OS patching and scaling
  • Development speed matters more than infrastructure control

Pick SaaS when…

  • You need productivity tools (email, docs, collaboration)
  • No custom development — consume as-is
  • Subscription pricing preferred over build cost

Pick FaaS/Serverless when…

  • Event-driven, short-lived tasks (seconds or less)
  • Highly variable or unpredictable traffic patterns
  • Pay-per-execution cost model fits the workload
🎯 Exam tip: "lift-and-shift with minimal changes" → IaaS. "deploy code without managing servers" → PaaS. "use email and Office apps" → SaaS.
This decision framework is exactly what exam scenario questions test. Coach attendees to look for trigger words: "lift-and-shift" or "migrate existing VMs" signals IaaS; "deploy web application" or "no OS management" signals PaaS; "email, collaboration, productivity" signals SaaS; "event-driven" or "scale to zero" signals FaaS. These keywords appear almost verbatim in real exam questions.

Lesson 03 · Scenario

A retail company has an existing order-processing application running on Windows Server 2019 on-premises. The application relies on a third-party COM component that requires direct OS access. Leadership wants to move it to Azure quickly with minimal code changes. Which service model should you recommend?

A IaaS — deploy the application on an Azure Virtual Machine running Windows Server 2019
B PaaS — deploy the application to Azure App Service on a Windows plan
C SaaS — migrate the order data to Microsoft Dynamics 365
D FaaS — rewrite the order-processing logic as Azure Functions triggered by a queue
✓ A is correct — IaaS (Azure VM) provides full OS-level control, supports the existing Windows Server 2019 environment, and enables lift-and-shift migration with minimal code changes. The third-party COM component requires direct OS access, which only IaaS provides.
  • B — App Service is PaaS and abstracts OS access; COM components requiring direct OS interaction are not supported in the sandboxed App Service environment
  • C — Dynamics 365 is a SaaS CRM/ERP product; it cannot host or run a custom order-processing application
  • D — Rewriting as Azure Functions requires significant code changes and architectural redesign, contradicting the "minimal changes" requirement
Give attendees 30–45 seconds to read the scenario before revealing options. The two key triggers in this scenario are "third-party component requiring direct OS access" and "minimal code changes" — both point unambiguously to IaaS. Reinforce that whenever a question mentions OS-level dependencies or lift-and-shift, IaaS is almost always the answer. Option D is the most tempting distractor for developers — clarify that serverless requires re-architecture, not migration.

Module 01 · Cloud Concepts

Public Cloud

Infrastructure owned and operated by a third-party provider — shared across many customers, delivered over the internet.

💰

No CapEx

Zero upfront hardware cost — pay only for resources consumed, billed by the second or minute

📏

Elastic Scale

Spin up global capacity in minutes; release it just as fast — the provider absorbs demand spikes

🛠️

Managed by Provider

Microsoft maintains physical hardware, networking, and hypervisor — you manage what you deploy

🎯 Exam tip: Public cloud = no hardware ownership by the customer. Azure, AWS, and Google Cloud are all public cloud providers.
Most AZ-900 candidates already interact with public cloud daily. Stress that "public" refers to the ownership model, not that data is publicly readable — access is controlled, but infrastructure is shared. Azure's 60+ regions are all public cloud.

Module 01 · Cloud Concepts

Private & Multi-Cloud

Private Cloud

  • Infrastructure dedicated to one organisation
  • Hosted on-premises or in a colocation facility
  • Full control over hardware, OS, and policies
  • Higher CapEx — organisation buys and maintains hardware
  • Required when data must never leave your facility
  • Azure Stack HCI / Hub brings Azure services on-prem

Multi-Cloud

  • Workloads distributed across two or more providers
  • Avoids vendor lock-in; use best-of-breed services
  • Increases operational complexity and cost management challenges
  • Common: Azure + AWS or Azure + Google Cloud
  • Azure Arc manages resources across clouds from one pane
  • Requires unified security and governance strategy
Private cloud is popular in heavily regulated industries — defence, healthcare, banking. Azure Stack lets those customers run Azure APIs on their own hardware. Multi-cloud is increasingly common in enterprises that acquired companies using different providers.

Module 01 · Cloud Concepts

Hybrid Cloud & Azure Arc

Hybrid cloud links on-premises infrastructure with public cloud — workloads can move between environments based on policy, cost, or compliance needs.

🔗

Connected Environments

On-premises datacentre connects to Azure via VPN or ExpressRoute — consistent identity, networking, and policy

🗂️

Data Sovereignty

Keep sensitive data on-prem while bursting compute-intensive workloads to Azure when demand spikes

🌐

Azure Arc

Extend Azure management, RBAC, and Policy to on-premises servers, Kubernetes clusters, and other clouds

🎯 Exam tip: Azure Arc = "project Azure management plane outward." It does NOT move workloads — it brings Azure governance to wherever the workload already lives.
Hybrid is the most common real-world pattern for enterprises mid-migration. Emphasise that hybrid is not a permanent destination — many organisations use it as a stepping stone to full public cloud adoption over 3-5 years. Arc is a hot exam topic.

Lesson 04 · Scenario

A financial services company processes customer transactions on-premises to satisfy a regulatory requirement that all payment data must remain within its own datacentre. During quarter-end reporting, processing demand spikes sharply and the on-premises hardware cannot keep up. The company wants to offload the reporting workload to Azure while keeping transaction data on-premises. Which deployment model best meets these requirements?

A Public cloud — migrate all workloads to Azure and enforce data controls using Azure Policy
B Private cloud — expand on-premises hardware capacity to handle peak demand
C Hybrid cloud — keep transaction data on-premises and burst reporting workloads to Azure
D Multi-cloud — split workloads between Azure and AWS to avoid any single provider dependency
✓ C is correct — Hybrid cloud is purpose-built for this pattern: sensitive data stays on-premises to meet the regulatory requirement while elastic Azure capacity absorbs the quarter-end spike, avoiding costly on-premises over-provisioning.
  • A — Moving all workloads to public cloud violates the regulatory requirement that payment data must remain in the company's own datacentre; Azure Policy enforces rules within Azure but cannot keep data physically on-premises
  • B — Expanding on-premises hardware adds significant CapEx and leaves that capacity idle for most of the year; it does not leverage cloud elasticity at all
  • D — Multi-cloud addresses vendor diversity, not the requirement to keep data on-premises; it also increases operational complexity without solving the core compliance constraint
Give attendees 30-45 seconds before revealing the answer. Emphasise that the regulatory constraint is the deciding factor — it rules out A immediately. Then distinguish B (pure private, no cloud) from C (hybrid). D is a common distractor because it sounds like a balanced answer, but multi-cloud is about provider diversity, not on-premises data residency.

Module 02 · Azure Architecture & Services

The Infrastructure Hierarchy

Azure's global footprint is organised into a four-level hierarchy — each layer adds a scope of isolation, redundancy, or compliance control.

🌍

Geography

A discrete market (e.g. United States, Europe) preserving data residency and compliance boundaries

🗺️

Region

A set of datacenters deployed within a latency-defined perimeter; 60+ regions across 140+ countries

🏢

Availability Zone

One or more physically separate datacenters within a region — independent power, cooling, and networking

🖥️

Datacenter

The physical facility housing servers, storage, and networking hardware that backs every Azure service

Module 02 · Azure Architecture & Services

Region Pairs & High Availability

What is a Region Pair?

  • Every Azure region is paired with another region in the same Geography, ≥300 miles apart
  • Paired regions receive updates sequentially — never simultaneously
  • If a platform outage affects a region, failover prioritises its pair
  • Some services (e.g. Azure Storage GRS) replicate automatically to the paired region
  • Example pairs: East US ↔ West US, UK South ↔ UK West

Availability Zones — Key Facts

  • Minimum of 3 AZs per enabled region (e.g. East US, West Europe)
  • Each AZ is isolated — a failure in AZ 1 does not affect AZ 2 or AZ 3
  • Zonal services are pinned to a specific AZ (e.g. a VM in Zone 2)
  • Zone-redundant services spread automatically across all AZs (e.g. Zone-redundant Storage)
  • Inter-AZ latency is typically under 2 ms — transparent to applications
🎯 Exam tip: AZs protect against datacenter-level failures; region pairs protect against region-wide failures. These are complementary, not interchangeable.
Students often confuse AZs with region pairs. Reinforce the distinction: AZs = same city, same day, millisecond latency; region pairs = hundreds of miles apart, disaster recovery. Both are needed for a comprehensive HA strategy. The exam tests which layer to apply for which failure scenario.

  • Azure China — operated by 21Vianet under Chinese law; separate Azure account required; some global services unavailable

The Four-Level Hierarchy

Every Azure resource lives inside a precise four-level container hierarchy — each level inherits policies and permissions from the level above.

🏢

Management Groups

Containers for multiple subscriptions — apply policies and RBAC across your entire organisation

💳

Subscriptions

Billing boundary and trust boundary for Azure AD — every resource belongs to exactly one subscription

📁

Resource Groups

Logical containers that hold related resources sharing the same lifecycle — delete the group to delete everything inside

⚙️

Resources

Individual Azure services — VMs, storage accounts, databases, and more; always belong to one resource group

Walk through each level top-to-bottom. A key exam concept is inheritance: a policy applied at the management group level flows down to every subscription, resource group, and resource beneath it. This "assign once, enforce everywhere" pattern is how enterprises manage hundreds of subscriptions consistently.

Module 02 · Azure Architecture & Services

Subscriptions & Resource Groups in Depth

Subscriptions

  • Unit of billing — usage rolled up per subscription
  • Separate subscriptions per environment (prod / dev) isolates costs and access
  • Soft limits apply per subscription (e.g. 980 resource groups max)
  • Trusts exactly one Azure AD / Entra ID tenant
  • Can be transferred between tenants or billing accounts

Resource Groups

  • Resources share a lifecycle — deploy, update, and delete together
  • Region is the metadata location, not the resource location
  • Resources in a group can span multiple regions
  • RBAC and policies can be scoped to a single resource group
  • A resource can only belong to one resource group at a time
🎯 Exam tip: a resource group's region stores the metadata about the group — resources inside can still be in any Azure region.
Candidates often confuse the resource group region with the region of the resources it contains. Stress the lifecycle angle — grouping by lifecycle (not by resource type) is the recommended pattern. For example, put a web app and its database in the same group if they are always deployed and retired together.

Virtual Networks & Subnets

An Azure Virtual Network (VNet) is an isolated, private network in Azure — the foundation for all Azure networking. Resources inside a VNet communicate securely by default.

🌐

VNet

Scoped to a single region; defined by an address space (e.g. 10.0.0.0/16) using CIDR notation

🔲

Subnet

Subdivides the VNet address space (e.g. 10.0.1.0/24); resources in different subnets can communicate within the same VNet

Connects two VNets (same or different regions) with low-latency, private traffic — no gateway required

Module 02 · Networking Services

Hybrid Connectivity: VPN Gateway vs ExpressRoute

VPN Gateway

  • Encrypted tunnel over the public internet
  • Site-to-site or point-to-site options
  • Up to ~10 Gbps (VpnGw5 SKU)
  • Lower cost; fast to provision (~45 min)
  • Best for: branch offices, remote workers, dev/test

ExpressRoute

  • Dedicated private connection — bypasses the internet
  • Up to 100 Gbps bandwidth
  • SLA-backed, consistent latency
  • Higher cost; weeks to provision
  • Best for: regulated industries, large data migrations
🎯 Exam tip: ExpressRoute does not traverse the public internet — it uses a connectivity provider's private circuit. VPN Gateway traffic is encrypted but does traverse the internet.
This comparison is one of the most tested networking scenarios on AZ-900. The key decision drivers are: compliance (regulated data must stay off the internet → ExpressRoute), cost sensitivity (VPN is cheaper), and bandwidth needs. A company can use both together for ExpressRoute failover.

🔍

Hosts DNS zones using Azure's global anycast network. Supports public zones (internet-facing) and private DNS zones (name resolution inside a VNet only)

Module 02 · Azure Architecture & Services

Blob Access Tiers

Blob Storage lets you assign a lifecycle tier per object — balancing storage cost against retrieval cost and latency.

  • Hot — Frequently accessed data (daily reads). Highest storage cost, lowest access cost. Default for new accounts.
  • Cool — Infrequently accessed; minimum 30-day retention. Lower storage cost, higher access cost than Hot.
  • Cold — Rarely accessed; minimum 90-day retention. Lower cost than Cool; data available within milliseconds.
  • Archive — Long-term retention; minimum 180 days. Lowest storage cost. Data is offline — must rehydrate (up to 15 hours) before reading.
  • Tiers can be set at the account level (Hot or Cool) or per-blob (Hot, Cool, Cold, Archive)
  • Archive-tier blobs cannot be read directly — rehydration copies data to Hot or Cool first
  • Lifecycle management policies automate tier transitions (e.g. move to Cool after 30 days, Archive after 90)
  • Early deletion fees apply if a blob is deleted or moved before its minimum retention period
🎯 Exam tip: Archive tier requires rehydration before data is accessible — it is not immediately readable like the other three tiers.
The exam frequently tests whether candidates know that Archive-tier blobs must be rehydrated — they are offline and cannot be read in place. Also emphasise that lifecycle policies are the recommended way to automate tier transitions rather than manual per-blob changes. Cold tier was added after Cool and sits between Cool and Archive in cost/latency trade-offs.

Redundancy Options

All storage accounts replicate data automatically. The option you choose determines where copies are stored and your durability SLA.

Within one region

  • LRS — Locally Redundant Storage
    3 copies within a single datacenter. Cheapest. No protection against datacenter failure. Durability: 11 nines.
  • ZRS — Zone-Redundant Storage
    3 copies across 3 Availability Zones in one region. Survives a full zone outage. Durability: 12 nines.
🎯 Exam tip: ZRS protects against a single AZ failure; LRS does not.

Across two regions (geo-redundant)

  • GRS — Geo-Redundant Storage
    LRS in primary region + asynchronous copy to secondary region. Secondary is read-only only after Microsoft initiates a failover. Durability: 16 nines.
  • GZRS — Geo-Zone-Redundant Storage
    ZRS in primary region + asynchronous copy to secondary region. Maximum durability and availability. Durability: 16 nines.
🎯 Exam tip: RA-GRS / RA-GZRS variants enable read access to the secondary region at all times — not just after failover.
The key differentiators are zone vs. region scope and whether read access to the secondary is always available. GZRS is the highest-durability option but also the most expensive — it is appropriate for mission-critical data that must survive both zone and regional failures. Remind students that the secondary region is always paired by Azure — you do not choose it.

🏢

Tenant

Module 02 · Azure Architecture & Services

Authentication vs Authorization

Authentication (AuthN) — Who are you?

  • Proves the identity of a user or service
  • Entra ID issues a token on successful sign-in
  • Strengthened by MFA — requires ≥2 verification factors
  • MFA factors: something you know, have, or are
  • SSO authenticates once to access many apps
  • Passwordless options: Windows Hello, FIDO2 keys, Authenticator app

Authorization (AuthZ) — What can you do?

  • Determines what an authenticated identity is permitted to do
  • In Azure, authorization is enforced by RBAC
  • Permissions granted via role assignments, not passwords
  • Always happens after authentication
  • Token claims carry scope and role information
  • Azure RBAC is separate from Entra ID directory roles
🎯 Exam tip: Authentication answers "who are you?" — authorization answers "what are you allowed to do?" The exam tests whether you can tell them apart in scenario questions.
Use a hotel analogy: showing your ID at check-in is authentication; your key card only opening your room is authorization. Stress that SSO reduces authentication fatigue and attack surface — fewer passwords means fewer credentials to steal.

Conditional Access & Zero Trust

Conditional Access policies evaluate signals at sign-in time and enforce access decisions — the engine that moves Azure toward a Zero Trust security model.

📍

Signals

User identity, device compliance, location (IP/country), app being accessed, real-time risk score

⚙️

Decisions

Allow, block, or require additional verification — e.g. step-up MFA or compliant device

🛡️

Zero Trust Principles

Verify explicitly · Use least-privilege access · Assume breach — never trust by network location alone

🎯 Exam tip: Conditional Access is an Entra ID P1/P2 feature — it is not available in the free tier. The AZ-900 exam expects you to recognise this as the tool for policy-driven access control.
Give a concrete example: a policy that blocks sign-ins from outside the US except when MFA is satisfied. Contrast this with old perimeter-based security where VPN location was enough. Zero Trust assumes the attacker is already inside the network, so every request must be verified regardless of origin.

Module 02 · Azure Architecture & Services

Role-Based Access Control (RBAC)

Azure RBAC controls what authenticated identities can do with Azure resources. A role assignment combines a security principal, a role definition, and a scope.

Built-in Roles (key four)

  • Owner — full access + manage access
  • Contributor — create & manage resources, no access management
  • Reader — view resources only, no changes
  • User Access Administrator — manage user access only

Scope Hierarchy

  • Management Group → Subscription → Resource Group → Resource
  • Permissions inherit downward — assignment at subscription applies to all its resource groups

Principle of Least Privilege

  • Grant only the permissions needed for the task — nothing more
  • Assign at the narrowest scope that satisfies the requirement
  • Prefer Reader over Contributor when writes are not needed
  • Use time-bound assignments via Privileged Identity Management (PIM) for admin roles
  • Regularly review and remove stale role assignments
🎯 Exam tip: RBAC is additive — if a user has Reader at subscription scope and Contributor on one resource group, the more-permissive role applies to that resource group.
Walk through a concrete assignment: assigning Reader to a contractor at resource group scope rather than subscription scope. This is the canonical least-privilege scenario the exam loves. Mention that deny assignments (via Azure Blueprints or Policy) override RBAC allow — a rare but exam-testable fact.

Azure Policy

Azure Policy evaluates resources against defined rules and enforces organisational standards automatically — at assignment time and on an ongoing basis.

📋

Policy Definition

A rule written in JSON that describes a condition and an effect (e.g. deny VMs without tags)

📦

🎯

Applies a definition or initiative to a scope: management group, subscription, or resource group

Walk through the lifecycle: you write (or pick a built-in) definition → bundle into an initiative → assign it to a scope. All resources within that scope are evaluated. Module 03 · Governance & Compliance

Policy Effects & Built-in Definitions

Policy Effects (in order of strictness)

  • Audit — flags non-compliant resources; no blocking
  • AuditIfNotExists — audits when a related resource is missing
  • Deny — blocks the create/update if it violates the rule
  • DeployIfNotExists — auto-deploys a missing companion resource
  • Modify — adds, updates, or removes properties/tags on resources
  • Disabled — definition exists but is not evaluated

Common Built-in Policies

  • Allowed VM SKUs — restrict to approved sizes
  • Allowed locations — enforce data residency
  • Require a tag on resource groups
  • Enable Microsoft Defender for Servers
  • Audit VMs without managed disks
  • ISO 27001 initiative (105+ policies bundled)
🎯 Exam tip: Deny stops non-compliant resources from being created. Audit only reports — it never blocks. You can also write custom policy definitions in JSON when no built-in fits.
The distinction between Audit and Deny trips up many candidates. Audit is a great starting point — deploy it first to understand your compliance baseline before switching to Deny. Custom definitions follow the same JSON schema as built-ins; the exam will not ask you to write JSON but may ask when to use custom vs built-in.

Module 03 · Governance & Compliance

Resource Locks & Tags

Resource Locks

  • CanNotDelete — read and modify allowed; delete is blocked for all users
  • ReadOnly — equivalent to Reader RBAC for everyone; no modifications or deletes
  • Applied at: management group, subscription, resource group, or individual resource
  • Child resources inherit a lock from their parent scope
  • Only the lock owner (or Owner/User Access Admin role) can remove a lock
  • Locks override RBAC — even an Owner cannot delete a CanNotDelete-locked resource without first removing the lock

Tags

  • Name/value metadata pairs attached to any Azure resource
  • Max 50 tags per resource or resource group
  • Tags are not inherited by default — child resources don't get parent tags automatically
  • Use cases: cost attribution, environment labels, owner tracking, automation triggers
  • Azure Cost Management can group and filter billing by tag value
  • Apply consistently with Azure Policy Modify effect or Require tag definitions
Locks and tags serve different purposes — locks are a safety net against accidental deletion or changes; tags are an organisational metadata system. A common exam gotcha: ReadOnly on a storage account prevents listing its keys (because that is treated as a write action). Also reinforce that tags do not flow down by default — you need policy to enforce consistent tagging.

Declare your desired infrastructure state in a file — Azure provisions it consistently every time. No manual clicks, no configuration drift.

  • Native Azure format — full resource coverage

The team uses Linux build agents and wants to version-control the infrastructure alongside application code. They need the simplest possible syntax with full Azure resource coverage. A Use the Azure portal to manually create each environment and export the resource configuration as a CSV
B Write Azure PowerShell scripts stored in a Git repository and run them on the Linux build agents
C Author Bicep templates stored in Git and deploy via Azure CLI in the pipeline
D Use ARM JSON templates deployed through the Azure portal's "Deploy a custom template" blade
✓ C is correct — Bicep provides simplified, version-controllable IaC with full Azure resource coverage; the Azure CLI runs natively on Linux and integrates cleanly into CI/CD pipelines. Idempotent deployments ensure each environment is identical.
  • A — manual portal clicks are not repeatable or version-controllable, and CSV exports are not deployable templates
  • B — PowerShell scripts are imperative, not declarative — they don't guarantee idempotency, and the Az module has limited support on Linux compared to the Azure CLI

Stress the two key decision factors in the scenario: Linux agents (favours CLI over PowerShell) and "simplest syntax with full coverage" (favours Bicep over raw ARM JSON). Module 03 · Monitoring Tools

Azure Monitor — The Central Hub

Azure Monitor collects, analyses, and acts on telemetry from your Azure resources, on-premises infrastructure, and applications — all in one place.

📊

Metrics

Numerical time-series data (CPU %, memory, request count) sampled every 60 seconds — stored for 93 days by default

📋

Logs

Structured or unstructured event records sent to a Log Analytics workspace and queried with Kusto Query Language (KQL)

🔔

Alerts

Notification rules fire when a metric or log condition is met — trigger emails, SMS, webhooks, or auto-remediation runbooks

🎯 Exam tip: Azure Monitor is the umbrella service — Log Analytics and Application Insights feed data into it; alerts and dashboards consume data from it.
Stress that Azure Monitor is not a standalone portal tab but a platform that unifies data from every other service. The 93-day metric retention limit is a favourite exam distractor — logs in Log Analytics can be retained much longer (up to 2 years by default, configurable further). KQL syntax questions do not appear on AZ-900, but learners should know KQL is what you use.

Module 03 · Monitoring Tools

Log Analytics & Application Insights

Log Analytics Workspace

  • Central store for log data from VMs, containers, network, and security sources
  • Query logs with KQL — filter, aggregate, join across tables
  • Retention: 30 days free; up to 730 days configurable
  • Feeds Azure Sentinel (SIEM) and VM Insights
  • One workspace can serve multiple subscriptions

Application Insights

  • Application Performance Monitoring (APM) for web apps and microservices
  • Tracks request rates, failure rates, dependency calls, and page load times
  • Automatic dependency mapping (Application Map)
  • Smart Detection — AI-powered anomaly alerts with zero configuration
  • Supports .NET, Java, Node.js, Python, and more via SDKs or auto-instrumentation
🎯 Exam tip: Log Analytics = infrastructure and platform logs; Application Insights = application-level telemetry. Both are features within Azure Monitor.
A common misconception is that Application Insights is a separate product. Since 2021 it is fully integrated into Azure Monitor as a "workspace-based" resource backed by Log Analytics. For the exam, distinguish the audience: ops teams use Log Analytics for VM and network logs; developers use Application Insights to debug slow transactions and trace exceptions end-to-end.

Module 03 · Monitoring Tools

Azure Advisor & Service Health

Azure Advisor

  • Free, personalised cloud consultant — analyses your actual usage and config
  • Cost: identify idle VMs, right-size over-provisioned resources, switch to reserved instances
  • Security: surface gaps in Microsoft Defender coverage and RBAC posture
  • Reliability: add redundancy, enable backups, fix single points of failure
  • Performance: recommendations for SQL, CDN, and App Service throughput
  • Operational Excellence: deployment health, monitoring coverage gaps

Azure Service Health

  • Azure Status: global status page — large-scale incidents affecting all customers
  • Service Health: personalised view of outages impacting your subscriptions and regions
  • Resource Health: per-resource availability history — was this VM affected?
  • Planned maintenance alerts and retirement notifications
  • Configure health alerts to notify your team via email or webhook
🎯 Exam tip: Advisor gives proactive recommendations on existing resources; Service Health gives reactive status during outages and advance notice of planned changes.
Walk through the five Advisor pillars — the exam expects students to know all five categories and what each one covers. For Service Health, the three-level hierarchy (Azure Status → Service Health → Resource Health) is a common exam question. Emphasise that Service Health alerts require manual configuration — they are not on by default.

Lesson 04 · Scenario

A development team is running a multi-tier web application on Azure App Service and Azure SQL Database. The team wants to automatically detect unusual spikes in failed HTTP requests and receive email notifications without writing custom alert queries. Which Azure service should they enable?

A Azure Monitor Metrics alerts on App Service request count
B Application Insights with Smart Detection enabled
C A Log Analytics workspace with a custom KQL alert rule
D Azure Advisor performance recommendations
✓ B is correct — Application Insights Smart Detection uses machine learning to automatically identify anomalies (including abnormal failure-rate spikes) and sends email notifications with zero manual query configuration.
  • A — Metrics alerts can fire on request count but require a manually defined static or dynamic threshold; they do not automatically learn normal patterns or distinguish failures without custom logic
  • C — A Log Analytics custom KQL rule can detect failure spikes, but it requires the team to write and maintain the query — the scenario explicitly asks for a no-query-writing solution
  • D — Azure Advisor provides proactive recommendations on configuration and cost but does not monitor live request failure rates or send real-time anomaly alerts
Give the group 30–40 seconds to decide before revealing the answer. The key discriminator is "without writing custom alert queries" — that phrase points directly to Smart Detection. Use this moment to reinforce why Application Insights exists as an APM tool on top of raw metrics: it understands application semantics and can establish baselines automatically.
